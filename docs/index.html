Methodology

how the desk computes

CSV format

The analyser expects a header row followed by one ticket per row. Column names are matched case-insensitively.

  • id — ticket identifier (any string)
  • created or opened or created_at — creation datetime
  • resolved or closed or resolved_at — resolution datetime (blank for open tickets)
  • priority — P1, P2, or P3
  • category — e.g. POS, Network, Email, Hardware, Access
  • store — store or site identifier

Datetime format: YYYY-MM-DD HH:MM or ISO 8601. Quoted fields containing commas are handled correctly.

Computed metrics

  • resolution_hours: (resolved − created) in decimal hours, per ticket
  • SLA compliance: a ticket meets SLA if resolution_hours ≤ the configured target for its priority. Open tickets (blank resolved) are excluded from all compliance and MTTR figures.
  • MTTR: mean time to resolve = average resolution_hours across all resolved tickets in the group
  • Repeat incidents: grouped by category (case-insensitive, trimmed). Cumulative resolution hours = sum of resolution_hours for that category.
  • Heatmap: counts tickets by day of week (Mon–Sun) and hour of day (00–23) using the created datetime. Open tickets are included in heatmap counts.

SLA defaults

  • P1: 4 hours
  • P2: 8 hours
  • P3: 24 hours

Edit the targets on the Data Desk before running the numbers.

The column (AI summary)

The button sends only the computed metrics string (never raw CSV rows) to the serverless backend, which calls NVIDIA-hosted DeepSeek or MiniMax. The model is instructed to write 3–5 sentences naming the top 3 operational risks and a recommended intervention for each.

With no API key configured, the column falls back to a deterministic summary generated entirely in the browser from the same metrics.

Data handling

Your CSV is processed entirely in the browser. Raw ticket rows are never sent to any server. The only external request is the optional column, which transmits only the computed metrics string (under 1,500 characters).
